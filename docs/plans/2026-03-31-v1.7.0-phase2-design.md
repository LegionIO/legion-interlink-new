# Legion Interlink v1.7.0 Phase 2: Proactive Intelligence

Design doc for three interconnected features that make interlink a living participant rather than a passive request-response tool.

## Context

Phase 1 (completed) added:
- 18 daemon IPC proxies for v1.7.0 endpoints
- Config schemas for 7 new daemon LLM settings
- LLM Pipeline settings panel (provider layer, context curation, debate, prompt caching, token budget, tier routing, escalation)
- GAIA presence indicator in sidebar
- Token usage display on assistant messages
- Pipeline insights rendering (debate + curation enrichments)

Phase 2 adds the agentic layer: the daemon can now initiate conversations, not just respond to them.

---

## Feature 1: Proactive Messaging

### What
GAIA's ProactiveDispatcher (legion-gaia v1.7.0) pushes unsolicited messages — dream cycle insights, observation-driven suggestions, bond-state reflections. Interlink needs a receive path.

### Daemon-side behavior
- `ProactiveDispatcher` rate-limits delivery: configurable frequency cap, cooldown between messages, ignore cooldown after user dismissal
- Messages carry intent metadata from `IntentClassifier` (7 types: direct_engage, share_insight, check_in, offer_help, express_concern, celebrate, reflect)
- Delivery gated by `PresenceEvaluator` — daemon knows if user is online/offline and whether they just transitioned online
- Dream-to-proactive bridge queues intents from dream cycle for waking dispatch

### Design: layered delivery

Three simultaneous delivery mechanisms ensure nothing is lost:

**Layer 1: Toast notification (always fires)**
- Every proactive message triggers a system-level toast via Electron's `Notification` API
- Toast shows: intent icon + first ~80 chars of message
- Click action: navigate to the GAIA thread (Layer 3)
- If app is backgrounded/minimized: OS notification with dock badge

**Layer 2: Inline injection (when user is in an active conversation)**
- If the user has an active conversation open and is not mid-composition, inject the proactive message as a visually distinct system message
- Visual treatment: subtle accent left-border (using GAIA's amber theme color), "GAIA" label badge, slightly muted background
- Message is non-interactive (no reply threading) — it's an observation, not a conversation turn
- If user IS mid-composition (typing), defer to toast only (don't interrupt flow)

**Layer 3: Dedicated GAIA thread (persistent log)**
- A pinned "GAIA" conversation in the sidebar (always at top, distinct from user conversations)
- Accumulates ALL proactive messages chronologically with their intent metadata
- Acts as a historical log of GAIA's autonomous activity
- User can scroll back to see past insights, check-ins, reflections
- Not a normal conversation — no compose input by default, but user can "reply" to engage GAIA about a specific insight

### Data flow
```
Daemon SSE → 'proactive-message' event
  → electron/ipc: new handler extracts { intent, content, metadata }
  → Main process:
      1. Emit Electron Notification (Layer 1)
      2. Forward to renderer via IPC broadcast
  → Renderer:
      1. If active conversation open + not composing → inject inline (Layer 2)
      2. Always append to GAIA thread store (Layer 3)
      3. Update GAIA thread unread badge in sidebar
```

### New components needed
- `electron/ipc/proactive.ts` — IPC handler for proactive message events
- `src/components/conversations/GaiaThread.tsx` — dedicated GAIA thread view (pinned, distinct styling)
- `src/components/thread/ProactiveMessage.tsx` — inline proactive message component (Layer 2)
- `src/lib/notifications.ts` — Electron notification wrapper with click routing
- Extend `ConversationList.tsx` — pin GAIA thread at top, show unread badge
- Extend `RuntimeProvider.tsx` — GAIA thread message store (separate from regular conversations)

### Config additions
```typescript
proactiveMessaging: {
  enabled: boolean            // default true
  toastEnabled: boolean       // default true
  inlineEnabled: boolean      // default true
  gaiaThreadEnabled: boolean  // default true
  deferWhileComposing: boolean // default true
}
```

## Feature 2: Trigger-Initiated Agentic Flows

### What
LegionIO v1.7.0 adds `Legion::Trigger` with webhook sources (GitHub, Linear, Slack). When a trigger fires, interlink should let GAIA autonomously triage and work the event.

### Daemon-side behavior
- `Legion::Trigger::Sources::Github` — receives GitHub webhooks (issues, PRs, pushes, comments)
- `Legion::Trigger::Sources::Linear` — receives Linear webhooks (issues, comments, status changes)
- `Legion::Trigger::Sources::Slack` — receives Slack events (messages, reactions, mentions)
- Each trigger produces a `Trigger::Envelope` with: source, event_type, payload, timestamp
- Envelopes are published to the SSE event stream

### Design: GAIA-driven autonomous triage

```
Trigger fires → SSE event → Interlink receives envelope
  → GAIA Triage (background):
      1. Classify: is this actionable or noise?
         - Noise (spam PR, duplicate issue, irrelevant chatter) → log + dismiss
         - Actionable → continue
      2. Create background conversation thread (sub-agent style)
      3. GAIA begins autonomous work:
         - Reads the payload context (issue body, PR diff, Slack thread)
         - Formulates a plan
         - Executes using available tools (shell, file access, web fetch, etc.)
      4. If GAIA needs human input → interrupt notification:
         "GAIA is working on GitHub issue #42 and needs your input"
         → Toast + sidebar badge, click navigates to the working thread
      5. If GAIA resolves autonomously → summary notification:
         "GAIA handled GitHub issue #42 — pushed fix to branch fix/auth-staging"
         → Toast + entry in GAIA thread (Feature 1, Layer 3)
```

### Triage classification
GAIA uses the daemon's LLM pipeline to classify incoming triggers:
- **Ignore**: bot-generated events, duplicate notifications, events from ignored repos/channels
- **Observe**: interesting but not actionable — log to GAIA thread for awareness
- **Act**: requires investigation or work — spin up background thread

User can configure triage rules:
```typescript
triggers: {
  enabled: boolean                    // default true
  autoTriage: boolean                 // default true
  triageModel: string                 // default '' (use default model)
  rules: Array<{
    source: 'github' | 'linear' | 'slack'
    eventType: string                 // e.g., 'issues.opened', 'message'
    action: 'ignore' | 'observe' | 'act'
    filter?: string                   // optional regex on payload
  }>
  maxConcurrentWorkflows: number      // default 3
  requireApprovalForActions: boolean  // default false (true = GAIA proposes, waits for approval)
}
```

### New components needed
- `electron/ipc/triggers.ts` — SSE event listener for trigger envelopes, triage dispatch
- `src/components/conversations/TriggerThread.tsx` — background workflow thread (sub-agent visual style but trigger-initiated)
- `src/components/conversations/TriggerBadge.tsx` — sidebar indicator for active trigger workflows
- Extend GAIA thread (Feature 1) — trigger events logged here when classified as "observe"
- Extend settings panel — trigger rules configuration UI

### Interaction with Feature 1
- Trigger outcomes (both autonomous resolutions and "needs input" interrupts) flow through the proactive messaging system
- GAIA thread becomes the central log for both proactive insights AND trigger activity
- Toast notifications use the same notification system

## Feature 3: Conditional Message Chain Architecture

### What
legion-llm v0.6.0 introduces a parent-link message chain model with sidechains and agent_id per message. Interlink has its own `messageTree` branching. When connected to the daemon, use its richer model; when standalone (mastra backend), keep the existing one.

### Current interlink model
- `messageTree`: `Record<string, string[]>` mapping parentId → childIds
- `headId`: current position in the tree
- `parentId` on each `StoredMessage`
- Branch navigation via `useBranchNav()` hook
- No sidechain concept, no agent_id

### legion-llm chain model
- `id` (UUID), `parent_id`, `sidechain` (boolean), `message_group_id`, `agent_id` per message
- `build_chain(conversation_id, include_sidechains:)` — reconstructs ordered history from parent links
- `sidechain_messages(conversation_id, agent_id:)` — queries background/sub-agent messages
- `branch(conversation_id, from_message_id:)` — creates new conversation from branch point
- `store_metadata` / `read_metadata` — tail-window session metadata

### Design: backend-conditional model

```typescript
// In RuntimeProvider.tsx, at conversation load time:
const backend = await app.agent.appStatus(); // 'legion-daemon' | 'mastra'

if (backend.type === 'legion-daemon') {
  // Use daemon's chain model — messages come with parent_id, sidechain, agent_id
  // Store these fields on StoredMessage
  // Branch navigation delegates to daemon's build_chain
  // Sidechains rendered as collapsible sub-threads
} else {
  // Use existing messageTree model — no changes
}
```

### StoredMessage extensions (daemon mode)
```typescript
type StoredMessage = ThreadMessageLike & {
  id: string;
  parentId: string | null;
  tokenUsage?: TokenUsageData;
  // Daemon chain fields (only populated in daemon mode)
  sidechain?: boolean;
  messageGroupId?: string;
  agentId?: string;
};
```

### Sidechain rendering
- In daemon mode, messages with `sidechain: true` are grouped by `agentId` and rendered as collapsible sub-threads within the main conversation
- Visual treatment: indented, lighter background, agent label badge
- Similar to how SubAgentInline already works, but driven by daemon chain data instead of interlink's sub-agent system

### Migration path
- No migration needed — daemon mode uses daemon's fields, mastra mode uses existing fields
- Both coexist on the same StoredMessage type (optional fields)
- If a conversation was started in mastra mode and the daemon comes online, continue with mastra model for that conversation (don't mix)
- New conversations auto-detect backend at creation time

### New/modified components
- Modify `RuntimeProvider.tsx` — conditional chain loading, sidechain grouping
- Modify `useBranchNav()` — delegate to daemon chain navigation in daemon mode
- `src/components/thread/SidechainGroup.tsx` — collapsible sidechain sub-thread rendering
- Modify `Thread.tsx` — render sidechains when present
- Extend conversation store — persist backend mode per conversation

### Config additions
```typescript
messageChains: {
  mode: 'auto' | 'interlink' | 'daemon'  // default 'auto' (auto-detect backend)
  showSidechains: boolean                  // default true
  collapseSidechainsByDefault: boolean     // default true
}
```

## Implementation Order

These features have dependencies:

```
Feature 1 (Proactive Messaging)  ← independent, build first
    ↓
Feature 2 (Trigger Flows)        ← depends on Feature 1 (notification system, GAIA thread)

Feature 3 (Message Chains)       ← independent, can parallel with Feature 1
```

Recommended sequence:
1. **Feature 1 + Feature 3 in parallel** — no shared code, different subsystems
2. **Feature 2 after Feature 1** — reuses notification system and GAIA thread

## Open Questions

1. Should the GAIA thread be a real conversation (persisted in conversations.json) or a separate store?
2. Should trigger triage happen in the daemon (via LLM pipeline) or in interlink's main process?
3. For message chains: should interlink ever write back to the daemon's chain store, or is it read-only?
